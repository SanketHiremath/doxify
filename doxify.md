# `doxify` — Doxygen Comment Adder for C/C++ Codebases

Add Doxygen-style function comments to `.h` and `.c` (or `.hpp`/`.cpp`) files,
leaving auto-generated or third-party code untouched. Runs a builder agent +
two reviewer agents per file pair, then commits. All agent rules are embedded
here — no external plugins required.

---

## Invocation

```
/doxify
/doxify src/foo
/doxify src/foo src/bar
```

- No argument — process every target file pair in the codebase that still lacks Doxygen blocks.
- One or more names (without extension) — restrict to those file pairs only.

---

## Scope

### Files to comment
| File | What gets documented |
|---|---|
| Public header (`.h` / `.hpp`) | Every **public** function prototype |
| Implementation (`.c` / `.cpp`) | Every **static** function **definition** |

### Files to NEVER touch
- Auto-generated files (e.g. files with generation headers, build artifacts)
- Third-party / vendored libraries (e.g. `third_party/`, `vendor/`, `extern/`)
- Files the user explicitly marks as off-limits

When in doubt about whether a file is auto-generated, read the top few lines —
generation notices ("DO NOT EDIT", "Generated by …") are the signal to skip.

---

## Doxygen Comment Format

Tag style: `@` (Javadoc). Block delimiter: `/** ... */`.

### Public function in `.h` / `.hpp`

```c
/**
 * @brief One-line summary of what the function does.
 *
 * Optional second sentence when the brief alone is not enough.
 *
 * @pre  pHandle must have been initialised by FooInit() before calling this.
 * @post Internal state is READY; FooProcess() may now be called.
 *
 * @tparam T  Element type; must be trivially copyable.
 *
 * @param[in]     paramA  Caller-provided value; function only reads it.
 * @param[out]    paramB  Function writes through this pointer; caller reads result.
 * @param[in,out] paramC  Function both reads and updates the pointed-to value.
 *
 * @retval STATUS_OK      Operation completed successfully.
 * @retval STATUS_TIMEOUT No acknowledgement received within the deadline.
 * @retval STATUS_BUSY    Resource locked by another caller.
 *
 * @deprecated Use FooInitEx() instead; this function will be removed in v3.
 */
ReturnType FunctionName(TypeA paramA, TypeB *paramB, TypeC *paramC);
```

### Static function in `.c` / `.cpp`

```c
/**
 * @brief One-line summary.
 *
 * @param[in] param Description.
 */
static void FunctionName(Type param)
{
```

### Tag rules

| Situation | Tags used |
|---|---|
| `void` return, no parameters | `@brief` only |
| Has parameters | `@brief` + one `@param` per parameter |
| Non-`void` return | `@brief` + `@return` OR `@retval` (see below) |
| Pointer written by callee | `@param[out]` |
| Pointer read and written | `@param[in,out]` |
| `bool` return | `@return true if …, false otherwise.` |
| Enum / status-code return with 2+ named outcomes | `@retval` once per constant instead of `@return` |
| Known precondition caller must satisfy | `@pre` |
| State guaranteed after the call | `@post` |
| C++ template parameter | `@tparam` (one per template parameter) |
| Function is superseded | `@deprecated Brief note of replacement` |

**`@return` vs `@retval`:** use `@return` for free-text descriptions (scalar math results, pointers, booleans). Use `@retval` when the function returns named constants from an enum or `#define` set and each value has a distinct meaning — one `@retval` line per constant actually returned. Never mix both in the same block.

**Never add:** `@details`, `@note`, `@warning`, `@file`, `@author`, `@date`.  
**Never document:** structs, enums, macros, typedefs, file-level blocks.

## Placement Rules

1. **Header public prototypes** — Doxygen block immediately above the prototype.
   No blank line between the closing `*/` and the function declaration.

2. **Source static definitions** — Doxygen block immediately above the `static ReturnType ...`
   line where the function body `{` begins. Do **not** comment forward declarations
   (the `static ReturnType FunctionName(...);` lines near the top of the file).

3. **Existing non-Doxygen blocks** — Any `/*****...*/` or `/*!` block above a function
   must be **replaced entirely**. Write a fresh Doxygen block from the signature;
   preserve zero text from the old block.

4. **All other existing comments** — leave completely untouched.

---

## Return Value Accuracy

Before writing `@return` or `@retval`, read the function body and identify the **actual** return values used.

- Read the return type definition (enum, typedef, `#define` set) in the codebase to understand what each constant means.
- Never substitute one constant for another visually similar one — similar names can have different values.
- For `bool`-returning functions: use `@return` and state the true/false condition explicitly.
- For enum / status-code returns: use `@retval` once per constant the function actually `return`s. If the function only ever returns one constant (e.g. always `STATUS_OK`), a single `@retval` line is correct. Do not fabricate `@retval` entries for constants the function never returns.

## Precondition and Postcondition Accuracy

Before writing `@pre` or `@post`, read the function body and any surrounding context:

- `@pre` — only add when there is a real, caller-visible precondition (e.g. a required init call, a non-NULL pointer, a locked mutex). Do not add trivial or self-evident preconditions.
- `@post` — only add when the function leaves the system in a state the caller needs to know about (e.g. a state machine transition, a resource now owned by the caller). Do not add obvious postconditions like "return value is set".
- If uncertain whether a precondition exists, read the function body for assertions, guards, or comments that imply it.

## Template Parameter Accuracy

`@tparam` applies only to C++ function templates and class templates. Do not add it to plain C functions or to C++ non-template functions. One `@tparam` per template parameter; include the constraint if one is documented in the code (e.g. "must be trivially copyable").

## Deprecation Accuracy

Add `@deprecated` only when the source code already signals deprecation — via a `[[deprecated]]` attribute, a `__attribute__((deprecated))`, a `DEPRECATED` macro, or an existing comment saying so. Do not invent deprecation notices. The `@deprecated` text must name the replacement if one is identifiable from the code.

## Execution Process

Work through file pairs **sequentially** (one pair at a time). For each pair:

1. **Build** — dispatch a builder agent (rules below)
2. **Spec review** — dispatch a reviewer agent to check compliance
3. **Quality review** — dispatch a reviewer agent to check accuracy
4. **Fix loops** — if either reviewer finds issues, send back to builder; re-review until clean
5. **Commit** — controller runs git after both reviews pass

The controller (main Claude session) handles **all git operations**.
Builder and reviewer agents must **never run git**.

## Agent Rules (embedded — no plugin required)

### Builder Agent Rules

When dispatching a builder agent via the `Agent` tool, prepend these rules
to the prompt so the agent behaves correctly even without the caveman plugin:

```
AGENT ROLE: doxify-builder
Caveman-ultra. Drop articles/filler. Code/paths exact, backticked. No narration.

SCOPE:
- 1 file ideal. 2 OK. 3+ → refuse with: "too-big. split: <n one-line tasks>."
- Edit existing files only (new file only if explicitly asked).
- No new abstractions, no drive-by refactors, no comment additions beyond the task.
- No Bash/shell available — cannot run git, cannot delete files.

WORKFLOW:
1. Read target file(s). Never edit blind.
2. Edit smallest diff that works.
3. Re-Read to verify the edit landed correctly.
4. Return a receipt.

OUTPUT RECEIPT FORMAT:
<path:line-range> — <change in ≤10 words>.
<path:line-range> — <change in ≤10 words>.
verified: <re-read OK | mismatch @ path:line>.

Diff is the artifact. Receipt is the proof. No exploration story.

REFUSALS (use these exact terminal lines):
- 3+ files → "too-big. split: <n one-line tasks>."
- Destructive op needed → "needs-confirm. op: <command>."
- Spec ambiguous → "ambiguous. ask: <one question>."

STATUS LINE (last line of every response):
DONE | DONE_WITH_CONCERNS | NEEDS_CONTEXT | BLOCKED
```

**Example builder dispatch prompt:**

```

 * @brief One-line summary.

```

```

Zero findings → "No issues."

BOUNDARIES:

End response with exactly one of:
✅ APPROVED
❌ ISSUES FOUND
```

## Spec Compliance Review Checklist

The spec reviewer prompt should ask for these checks specifically:

- Every public function prototype in the header has a `/** ... */` block
- Every static **definition** in the source has a `/** ... */` block
- No **forward declaration** in the source was commented
- No old `/*****...*/` or `/*!` block remains above any function
- Correct tags only: `@brief`, `@param[in/out/in,out]`, `@return`, `@retval`, `@pre`, `@post`, `@tparam`, `@deprecated`
- No `@details`, `@note`, `@warning` present
- `@return` / `@retval` absent on `void` functions, one or the other present on non-`void`
- `@return` and `@retval` not mixed in the same block
- `@tparam` present only on actual template functions/classes
- `@deprecated` present only where source code signals deprecation
- No struct / enum / macro / typedef comments added

## Quality Review Checklist

The quality reviewer prompt should ask for these checks specifically:

- `@brief` descriptions are accurate to what the function actually does
- `@param` descriptions are meaningful (not just the parameter name repeated)
- `@param` direction tags (`[in]`/`[out]`/`[in,out]`) match how the pointer is used
- `@return` text names the **actual constants or values** the function returns
  (read the function body — similar-looking constants may have different values)
- `@retval` entries exist only for constants the function **actually returns** — no fabricated entries
- `@pre` conditions are real caller-visible requirements, not self-evident trivialities
- `@post` conditions are state changes the caller needs to know about
- `@deprecated` text names the replacement if one exists in the code

## Commit Format

```bash
# Both header and source modified:
git add src/foo.h src/foo.c
git commit -m "docs: add Doxygen to foo.h and foo.c"

# Header only (no static functions in source):
git add src/foo.h
git commit -m "docs: add Doxygen to foo.h"
```

## Final Quality Gate (per file pair)

Before marking complete:

- [ ] Every public function prototype in the header has a Doxygen block
- [ ] Every static definition in the source has a Doxygen block
- [ ] No forward declaration in the source was commented
- [ ] No old `/*****...*/` or `/*!` block remains above any function
- [ ] `@return` / `@retval` uses the actual constants/values the function returns; not mixed in same block
- [ ] `@retval` has no fabricated entries for constants the function never returns
- [ ] `@pre` / `@post` added only where real preconditions/postconditions exist
- [ ] `@tparam` added only on actual template functions/classes
- [ ] `@deprecated` added only where source code signals deprecation; names the replacement if known
- [ ] No `@details`, `@note`, `@warning` tags present
- [ ] No struct / enum / macro / typedef comments added
- [ ] No file-level `@file` blocks added
- [ ] No logic, includes, or formatting changed
